## The critical gaps (ranked)

### 1. Security is absent — TR-MKP-SEC-001 (and TR-MKP-ARCH-001 authenticated IPC)
The load-bearing security requirement is unimplemented, and this is not a nuance — it is the whole layer.
- `MouseKeyProxyImpl.Pair` accepts a hardcoded `PairingCode == "valid-test"`, returns no cert, persists nothing (`src/MouseKeyProxy.Service/MouseKeyProxyImpl.cs:55-62`).
- No crypto/cert/TLS/secure-storage anywhere in `src/` (repo-wide grep for `X509`/`Certificate`/`AesGcm`/`SslCredentials`/`ChannelCredentials`/`Encrypt` matches only the proto message fields).
- Kestrel is plaintext h2c, `ListenAnyIP`, no `AddGrpc` interceptor, no authorization (`src/MouseKeyProxy.Service/Program.cs:26-32,57`).
- Every effect-bearing RPC (`InjectInput`, `OpenSession`, `SetMousePosition`, `SetFocusByHwnd`, `CaptureScreenshot`, `ClearModifiers`, `EmergencyRelease`) runs with no pairing/authorization check; `PeerId` is used only for logging.
- The agent control named pipe has no `PipeSecurity`/ACL and no token/identity check — any local process can drive `InjectInput`/`EmergencyRelease`/`CaptureScreenshot` (`src/MouseKeyProxy.Agent/AgentControlPipeServer.cs:73-132`).
- The only "security" test (TEST-MKP-005) certifies the stub.

Consequence: anyone who can reach the gRPC port can inject input / capture the screen; any local process can drive the agent. This is the gap that the shutdown-endpoint question surfaced.

### 2. Safety failsafes are missing — TR-MKP-RELI-001
For an exclusive input-forwarding tool these are safety-critical:
- No ClipCursor-release-within-2s timer; `Win32CursorClip.Release` is manual only (`src/MouseKeyProxy.Agent/Win32SeamImpls.cs:572-576`).
- No reconnect-give-up-within-5s logic (GrpcChannel defaults).
- No local-only fallback (grep finds none).
- No server-side cleanup on disconnect (`OpenSession` `finally` only logs, `MouseKeyProxyImpl.cs:118-121`).
- No dedicated emergency-release hotkey (the one hotkey is bound to toggle).
- seq/ack is cosmetic and off the live path (the forwarder uses unary `InjectInput`, `RemoteInputForwarder.cs:264`); the proto's mandated `VERSION_MISMATCH` handshake is never enforced.
- The one "failsafe" test is a microsecond in-memory simulation asserting `<2000ms` (`LatencyFailsafeHarness.cs:43-56`).

### 3. Clipboard sync does not exist end-to-end — FR-MKP-004
Real LIFO merger + DPAPI persistence exist as a library with genuine tests, but nothing captures the OS
clipboard and nothing sends/receives `ClipboardPush` over the wire (received frames are logged and
dropped). TR-MKP-CLIP-001 also omits sequence ordering, per-item size limit, and total-storage cap.

### 4. The test suite gives false confidence
111 green, but ~45% of it is source-scan or tautological, and several requirements are "verified" only by
scans or env-gated no-ops (failsafe timing, two-machine E2E, hardware HID, hotkey/ownership). The
Requirements-Matrix "Status" column means *traceability-tracked*, not *implemented* — there is no
authoritative build-status anywhere, which is precisely how a stubbed TR-MKP-SEC-001 stayed invisible.

Concrete danger case (found and FIXED in this pass): `ReplBidiConstructionTests.Repl_Main_Drives_Real_Path_No_Crash`
was a **unit** test that called `Program.Main(["inject-text", "real-frame-from-main"])`, which opened a real
gRPC channel to `localhost:50051` and — against any live local agent — injected the literal keystrokes
"real-frame-from-main" into the operator's foreground window on every test run. Its only assertion was the
tautology `Assert.True(code == 0 || code == 1)`. This is a live demonstration of TR-MKP-SEC-001 (unauthenticated
`InjectInput`) plus a unit/integration misclassification. Fixed by adding a `Program.TestGrpcClientFactory` seam
and rewriting the test to drive a mocked service (NSubstitute), asserting the dispatched `InjectInput` text and a
0 exit — hermetic, no socket, no OS injection. Remaining hygiene debt (not yet fixed): the large source-scan
files (`HotkeyRegistrationRegressionTests.cs`), the tautological `AgentControlProtocolTests` / `HotkeyToggleUnitTests`,
and the no-assert `LatencyFailsafeHarness`.

### 5. Pi appliance diverges from intent — FR-MKP-012 / TR-MKP-HID-001
The built PiHid is a standalone bearer-token HTTP service accepting only raw report bytes and three
hardcoded chords; it is not the intended gRPC-paired-peer-with-HID-injection-backend, and no on-hardware
gadget/enumeration has ever run.

## Second pass — traceability, build/CI, and code hygiene

### Requirements & traceability integrity

- **`Requirements-Matrix.md` is structurally broken:** header declares 3 columns; rows 7-58 carry 5; rows 59-67 (FR-MKP-012, TR-MKP-HID-001, TEST-MKP-023..029) carry only 3 with no Type and no trace-links — the newest HID/input-regression requirements have **zero recorded traceability** (`Requirements-Matrix.md:5,7,59-67`).
- **Matrix FR rows are stale vs the newer `TR-per-FR-Mapping.md`** — forward/reverse link disagreements on FR-HOTKEY-001, FR-MKP-001/002/003/005/012, FR-OWNERSHIP-001.
- **Every FR carries two divergent acceptance-criteria blocks** (a plain list and a checkbox list) with different item counts — an integrity hazard (FR-MKP-001/002/003/005).

- Positive: the azure/github wiki mirrors are byte-identical to canonical — no drift.

### Build / CI / release integrity
- **The repo's own `CLAUDE.md` is the wrong project's file** — a verbatim copy of McpServer's CLAUDE.md (references `McpServer.*`, `./build.ps1`, GitHub Actions `build.yml`, `ValidateTraceability`, ASP.NET Core 9). None of it applies to MouseKeyProxy. The primary agent-guidance file describes a different product; several claims elsewhere in this audit that "CLAUDE.md says X" trace to this mis-targeted file.
- **The Nuke `Test` target runs the WHOLE solution, integration tests included — no exclusion** (`build/Build.cs:77-82`, `MouseKeyProxy.slnx:18`). The CLAUDE.md claim that integration is excluded is false here.
- **CI correctness depends on live lab hardware.** `TransitionE2ETests` are ungated: `LabTopology.ResolvePeers()` throws on any machine not named `payton-legion2`/`payton-desktop` (`LabTopology.cs:27-28`), and `AssertGrpcReachable` opens a real `TcpClient` to `:50051` (`TransitionE2ETests.cs:8-14`). On the self-hosted `Default` agent (`azure-pipelines.yml:11-12`), CI is red off-lab or without both peers live; `scripts/publish-release.ps1:34` inherits the same exposure. (The HID E2E gate *is* env-guarded — `HardwareHidComplianceTests.cs:99-103` — so the gating discipline is inconsistent.)
- **No real traceability validation** — only brittle `Assert.Contains` substring checks in unit tests; the CLAUDE.md `ValidateTraceability` step does not exist.
- **CA1416 (platform compatibility) suppressed repo-wide** (`Directory.Build.props:9-10`) while projects deliberately span Windows and linux-arm64 — this hides the exact cross-platform-API-misuse it exists to catch, and it directly undermines the "Pi runs the Service" goal (a Windows-only API leaking into the Linux build would not be flagged).
- **Central Package Management disabled** (no `Directory.Packages.props`); versions drift: deprecated **Grpc.Core 2.46.6** (test-only) vs the 2.66.0 managed stack; **xunit v2 (2.9.3)** in PiHid.Tests vs **xunit.v3** everywhere else; `Microsoft.Extensions.*`/`System.*` at 9.0.0/8.0.0 on a net10 TFM.
- Positives: NuGet publish is robustly gated (tag `v*` + `NUGET_API_KEY` + HEAD==latest-tag); GitVersion drives all versions with no hardcoded version. No GitHub Actions workflow exists (CLAUDE.md claims one); `tests/MouseKeyProxy.WindowProbe` is a WinExe app, not a test project (cosmetic mislabel).

### Code hygiene & security smells
- **HIGH — the `"valid-test"` pairing code is the sole auth gate** before full remote input injection (`MouseKeyProxyImpl.cs:55`); it also leaks into e2e scripts. (= TR-MKP-SEC-001.)
- **`LabTopology.ResolvePeers()` throws on any non-lab machine** (`LabTopology.cs:27-28`) — the product only runs on the two named developer boxes; the agent falls back to hardcoded `http://payton-desktop:50051` (`Agent/Program.cs:1026`).
- **Overstated `[REAL … SUCCESS]` banners** after unacked one-way sends (`Agent/Program.cs:1106`; many `[REAL gRPC …]` in `Repl/Program.cs`) — they mean "sent without throwing," not remote success.
- **Swallowed errors:** empty `catch {}` in forwarder teardown (`RemoteInputForwarder.cs:302`), accept loop (`AgentControlPipeServer.cs:92`), pairing reset (`Agent/Program.cs:594`), Win32 op (`Win32SeamImpls.cs:325`); many failures reported only via `Debug.WriteLine` (silent in Release).
- **Sync-over-async on the WinForms tray thread** (`Agent/Program.cs:1105`, `RemoteInputForwarder.cs:323`) can hang the UI; fire-and-forget send loop whose faults are never observed (`RemoteInputForwarder.cs:99` + swallowed `:302`).
- `InputCommandHandler.cs:37` throws `NotSupportedException` for any Pi-HID text that is not one of three hardcoded chords.
- Positives: no committed secrets (Pi bearer token is env-var; the image provisioner fetches only a public SHA-verified image); the Pi HID service auth fails closed (`PiHidEndpointHandlers.cs:89-94`) — the opposite of the gRPC gate.

## Recommendation
Freeze feature work. The remediation order the evidence supports:

1. **Fix `CLAUDE.md`** — replace the McpServer copy with the real MouseKeyProxy build/test/CI facts. The project has been operating against another product's guidance.
2. **TR-MKP-SEC-001 + TR-MKP-ARCH-001**: real pairing key negotiation, persisted secret/cert material, a gRPC authorization interceptor, and authenticated agent IPC. This is the load-bearing gap and gates any destructive RPC (e.g. the parked `Shutdown`).
3. **De-risk CI**: gate or relocate `TransitionE2ETests` so a routine build does not depend on live lab hardware, and stop `LabTopology` from hard-failing off the two named machines.
4. **Restore build guardrails**: re-enable CA1416 (or scope its suppression) now that the code genuinely spans Windows + linux-arm64; adopt `Directory.Packages.props` (CPM) and reconcile the Grpc.Core/xunit version splits.
5. **TR-MKP-RELI-001 safety failsafes** (2s clip release, 5s reconnect give-up, server-side disconnect cleanup, emergency hotkey, local-only fallback).
6. **Requirements hygiene**: give the matrix a real implementation-status column, normalize its table, de-duplicate the twin acceptance-criteria blocks, and fix the dangling policy pointer — so this class of gap cannot hide behind a green-but-hollow test run again.
