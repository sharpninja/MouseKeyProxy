# Codex Peer Review: PLAN-MKP-001

**Reviewed plan**: `docs/PLAN-MKP-001.md`  
**Review date**: 2026-07-03  
**Reviewer**: Codex  
**Verdict**: Not approved for construction yet. The plan has a useful product direction, but it is not decision-complete enough for Byrd-style implementation. Several items are infeasible or ambiguous as written, and the required FR/TR/TEST traceability is incomplete.

## Scope Reviewed

- `docs/PLAN-MKP-001.md`
- `AGENTS-README-FIRST.yaml` planning rules, especially the Byrd and Planning Standard sections
- Current repo state: `global.json`, `.gitignore`, `docs/todo.yaml`, and git remotes

## Blocking Findings

### B1. Service/tray/input architecture is not decision-complete

**Evidence**: Plan lines 35, 42, 55-65, 157, and 209 describe a Windows service plus WinForms tray hybrid, but the initial file list puts `HotkeyService`, `InputHookService`, `ClipboardManager`, and `Tray` under one main service project.

Low-level keyboard/mouse hooks, hotkeys, clipboard listeners, cursor clipping, and foreground-window operations are user-session concerns. A Windows service runs in Session 0 and cannot be treated as the owner for interactive desktop input. The plan notes this risk, but it does not make the architectural decision that prevents implementation agents from putting input code in the wrong process.

**Required fix**: Add an ownership table before construction:

- Service process: networking, pairing state, persistence, watchdog/reconnect, service lifecycle.
- User-session agent/tray: hooks, hotkeys, clipboard listener, ClipCursor, SendInput, foreground/focus/window enumeration.
- Local IPC between tray/agent and service: named pipe, loopback gRPC, or another explicit mechanism with auth, timeout, reconnect, and failure semantics.
- Startup model: how the per-user tray/agent launches at login while the service starts at boot.

Do not begin implementation until each feature has a named owning process and IPC boundary.

### B2. "Proxy all keyboard input" is infeasible as written

**Evidence**: Plan lines 15, 116-122 require all keyboard input, including media keys, modifiers, Win keys, shortcuts, and a Ctrl+Alt+Del subset.

Windows intentionally blocks or constrains some inputs. Ctrl+Alt+Del/SAS cannot be generated by normal applications with `SendInput`. Secure desktop/UAC prompts, lock screen, some elevated-process scenarios, and UIPI integrity boundaries will not behave like ordinary user input. The current wording makes the acceptance criteria impossible to satisfy literally.

**Required fix**: Replace "all keyboard input" with a support matrix:

- Supported: ordinary keys, scan-code keys, modifiers, Win-key combinations where Windows permits them, media keys, mouse buttons/wheel, and text injection where applicable.
- Conditionally supported: elevated target windows, IME/dead-key layouts, keyboard layouts that differ across machines.
- Not supported: Ctrl+Alt+Del/SAS generation, secure desktop, lock screen, login screen, and any input blocked by Windows integrity/security policy.

Each unsupported case needs observable failure behavior and tests that assert the product does not hang, trap the user remotely, or claim success.

### B3. Byrd/TDD traceability is incomplete

**Evidence**: Plan lines 147-151 list only two example TEST records and explicitly defer full TEST/TR mapping. Lines 156-166 describe broad phases but do not name first failing tests, expected red state, green criteria, or exact validation commands. `AGENTS-README-FIRST.yaml` lines 249 and 251-255 require small gated slices, FR/TR/TEST records, coverage mapping, first tests, expected red states, green criteria, and exact validation scope before implementation.

The plan currently says it follows Byrd, but it does not yet satisfy the local planning contract. I also observed `docs/todo.yaml` contains only the header `# TODO items for this workspace`, so the plan's claim that requirement/TODO state is ready is not demonstrated in the repo-local artifacts.

**Required fix**: Before construction, add a traceability section with:

- FR-MKP IDs mapped to TR-MKP IDs and TEST-MKP IDs.
- A first-test table for each slice: test name, project/file, expected red failure, implementation owner, green criteria, and command.
- Gate commands such as `dotnet test <specific test project> --filter ...`, broader suite commands, and manual two-machine validation steps.
- TODO IDs for each gate, using the workspace's canonical uppercase ID format.

### B4. The REPL/global tool install story is not technically valid as stated

**Evidence**: Plan lines 17, 46, 65, and 139 say the global dotnet REPL tool carries or installs the service, and that `dotnet tool install ...` registers the service.

`dotnet tool install` installs the tool package; it does not run arbitrary post-install service registration. A global tool can expose a command that performs service installation after the user runs it, but the plan's acceptance criterion says installation itself registers the service. That will mislead implementers and tests.

**Required fix**: Decide and document the actual deployment contract:

- `dotnet tool install --global MouseKeyProxy.Repl`
- `mkp service install` or equivalent command copies/extracts the service payload, registers the Windows service, adds firewall rules, and validates status.
- `mkp service uninstall` reverses service and firewall changes.
- Define where service binaries live, how they are updated, how elevation is requested, and how failed partial installs roll back.

### B5. Pairing, TLS, discovery, and authorization are underspecified

**Evidence**: Plan lines 18, 22, 38-39, 131-145, and 210 mention UPnP/broadcast discovery, key negotiation, TLS, persisted secrets, and firewall rules, but do not define the security model.

This application can inject input, set focus, move the pointer, and read/write clipboard data. A vague "TLS + negotiated secrets" statement is not enough. The plan needs to lock identity, trust-on-first-use behavior, revocation, replay protection, and which peer is authorized to call high-impact RPCs.

**Required fix**: Add a pairing protocol section:

- Discovery transport and port selection.
- Manual confirmation or pairing code flow.
- Certificate/secret generation, storage scope, rotation, and revocation.
- Whether gRPC uses mTLS, bearer tokens, HMAC, or another concrete mechanism.
- Authorization matrix for `SendInput`, `InjectInput`, clipboard sync, `SetMousePosition`, `LocateProcess`, and `SetFocusByHwnd`.
- Tests for unpaired peer rejection, wrong secret rejection, revoked peer rejection, and firewall rule cleanup.

## High-Severity Findings

### H1. The protobuf contract is not ready to implement

**Evidence**: Plan lines 77-103 define the proto.

The proto omits several fields needed for reliable input proxying: protocol version, peer identity, sequence numbers, acknowledgement/retry semantics, error codes, correlation IDs, keyboard layout/scan-code semantics, mouse wheel/XButton events, Unicode/text input, display coordinate mode, and clipboard messages. `InputKind` also assigns `KEY_DOWN = 0`, which becomes the proto3 default for missing values; use an explicit `INPUT_KIND_UNSPECIFIED = 0` instead.

**Required fix**: Treat the proto as a design artifact, not a sketch. Add versioning, sequencing, failure codes, complete input event coverage, auth metadata expectations, and compatibility rules.

### H2. Clipboard LIFO design needs a real data model

**Evidence**: Plan lines 16, 40, and 124-129 require real-time clipboard sync, LIFO history merge, max 50 entries, and DPAPI encryption.

Windows clipboard is not a native LIFO stack. The plan needs to define what an entry is, which formats are supported, how duplicates are detected, how simultaneous copies on both machines merge, how loops are prevented, how binary/image/file/html content is handled, and how privacy opt-out works. DPAPI scope also matters: `CurrentUser` works for tray/user-session data; `LocalMachine` or service account storage has different security and portability properties.

**Required fix**: Add a clipboard model with schema, supported formats, dedupe key, merge algorithm, loop prevention, persistence location, DPAPI scope, size limits, and tests for concurrent updates.

### H3. Source-control instructions conflict with observed repo state

**Evidence**: Plan line 32 says Azure DevOps origin is primary and GitHub is downstream mirror. The current repo has only `origin` pointing to `https://github.com/sharpninja/MouseKeyProxy.git`.

This is not just naming. Automation and handoff agents will push/fetch from the wrong remote if the plan does not match reality.

**Required fix**: Either make GitHub `origin` the accepted source of truth for this repo, or add the Azure remote explicitly and name GitHub as `github`. Update the plan to match the actual remote contract.

### H4. The product scope says "exactly two systems" but the architecture expands to multi-remote

**Evidence**: Plan line 10 says exactly two Windows 11 systems. Lines 21, 45, and 133 require a host-central model with multiple remotes.

Multi-remote support changes pairing, addressing, UI, config shape, test matrix, Mirror Mode, clipboard conflict handling, and authorization. It is not free. If v1 is exactly two machines, keep the data model extensible but do not require multi-remote behavior in the first acceptance gate.

**Required fix**: Define v1 as either two-node only or true host-plus-many-remotes. If multi-remote is deferred, move it into a later FR/TODO with separate tests.

### H5. Verification is not executable

**Evidence**: Plan lines 199-200 say "See original" and list a manual flow without concrete commands.

This is not enough for a handoff plan. The plan must tell an implementation model exactly what to run, what machines are needed, what success looks like, and what evidence to capture.

**Required fix**: Replace the verification section with command-level gates:

- Restore/build/test commands.
- Service install/uninstall commands.
- REPL pairing commands.
- Manual two-machine smoke checklist.
- Evidence paths for logs, screenshots, service status, and encrypted clipboard store checks.

## Medium-Severity Findings

### M1. Tray actions invoking the REPL per command may be fragile

**Evidence**: Plan line 20 says tray actions invoke the REPL for gRPC sends.

If that means spawning the REPL process for every tray click, latency, quoting, timeout, and error handling will be poor. If it means sharing the same command implementation library, that is fine, but the plan needs to say so.

**Required fix**: Define whether tray calls a shared command library, a local service IPC endpoint, or an external `mkp` process. Include timeout and error-display behavior.

### M2. Window/process control endpoints need authorization and safety constraints

**Evidence**: Plan lines 23 and 82-98 include `LocateProcess`, `SetMousePosition`, and `SetFocusByHwnd`.

These operations expose local process/window information and can manipulate focus. They need authorization checks, input validation, and failure modes for hidden windows, elevated windows, stale HWNDs, and cross-desktop boundaries.

**Required fix**: Add endpoint-specific authorization and error semantics. Do not expose these as generic "trusted peer can do anything" calls without tests.

### M3. Failsafe behavior is missing

The product can clip the cursor and redirect input. The plan needs explicit local escape behavior for crashes, lost connection, bad hotkey config, service stop, tray exit, remote disconnect, and startup before pairing.

**Required fix**: Add a "local safety" section: emergency release hotkey, automatic ClipCursor release on process exit, timeout on remote-loss, local-only mode, and config reset command.

### M4. Reliability and event ordering are underspecified

The plan says "Reliable sequenced events" but does not define buffering, backpressure, dropped peer behavior, key-up recovery, stuck modifier cleanup, reconnect replay, or rate limiting. Input systems fail badly when key-up events are lost.

**Required fix**: Add an event transport policy with sequence numbers, ack windows, stuck-key recovery, disconnect cleanup, and tests that simulate lost key-up events.

### M5. Post-completion marketing and media assets should be separate scope

**Evidence**: Plan lines 26 and 202-206 include social drafts, logo/branding, video scripts, AIUnit visual review, and code review.

These are useful later, but they are not construction prerequisites for the Windows input/clipboard product. Keeping them in this plan increases scope noise and makes approval ambiguous.

**Required fix**: Move post-completion assets into a separate Transition or Release plan with its own TODOs.

## Strengths

- The hotkey-only/no-edge-crossing product constraint is clear and useful.
- The plan correctly calls out service-session limitations as a major risk.
- DPAPI, LocalAppData settings, REPL management, and service lifecycle are reasonable building blocks once ownership is clarified.
- The plan already separates peer review from implementation, which is the right control point for this level of risk.

## Required Revision Checklist

Before approving implementation, revise `PLAN-MKP-001.md` to include:

1. Process ownership and IPC architecture for service, tray/user agent, and REPL.
2. A realistic keyboard/input support matrix with explicit Windows security limitations.
3. FR/TR/TEST mapping and TODO IDs that satisfy the workspace Planning Standard.
4. Named first tests, expected red states, green criteria, and exact validation commands per Byrd gate.
5. A concrete install/update/uninstall contract for the global tool, service payload, elevation, and firewall rules.
6. Pairing/TLS/authorization design with negative security tests.
7. A complete proto/data-contract revision, including versioning, sequencing, error codes, and `UNSPECIFIED = 0` enums.
8. Clipboard schema, merge algorithm, format support, loop prevention, DPAPI scope, and concurrency tests.
9. Source-control remote policy that matches the actual repo.
10. A v1 scope decision: two-node only or true multi-remote.
11. Failsafe behavior for ClipCursor/input redirection, stuck keys, disconnects, and crashes.
12. Separation of product implementation scope from marketing/media post-completion tasks.

## Approval Recommendation

Hold construction. Approve only a revision pass on the plan plus requirements/traceability updates. Once the blockers above are resolved, the project can move into an Elaboration slice focused on mocks and risk prototypes for:

- user-session input agent vs service IPC,
- realistic keyboard event capture/injection limits,
- pairing/auth rejection paths,
- clipboard LIFO model,
- service install/uninstall lifecycle.
